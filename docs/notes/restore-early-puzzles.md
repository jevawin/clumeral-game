# Pick-up notes — restore original early puzzles

Tracking issue: **[#235](https://github.com/jevawin/clumeral-game/issues/235)** — "Restore original early puzzles — archive inaccurate after generator method change"

This is a hand-off doc to pick up the investigation locally. It records what was
already established (so you don't re-derive it) and the exact next steps.

---

## The problem

Early puzzles (from launch, #1 / `2026-03-08`, onward) were generated by an
**earlier CSV-based method** that no longer exists in git history. The current
on-the-fly generator recomputes those dates differently, so the archive
(`/puzzles`, `/api/puzzle/:num`, `/api/puzzle/:num/solution`) may show puzzles
that differ from what players originally saw.

## What's already been confirmed (don't redo)

- **Puzzles are KV-first.** `getDailyPuzzle()` in `src/worker/index.ts` reads the
  stored puzzle from the `PUZZLES` KV namespace by date key, and only recomputes +
  `put()`s if KV has no entry. KV entries have **no TTL** and are **never
  overwritten** once written.
- **Git history is squashed.** Root commit is `2026-04-01` (~50 commits). **No
  `.csv` was ever committed.** `EPOCH_DATE = 2026-03-08` = puzzle #1, ~3 weeks
  before history begins.
- The surviving `puzzle.js` → `src/worker/puzzle.ts` is **already the "no CSV
  required" rewrite** (per its own header comment). The prior CSV generator
  existed but was never committed.
- The April "architecture overhaul" (`c508a85`) was a **pure refactor** — old
  `puzzle.js` and current `src/worker/puzzle.ts` are algorithmically identical
  (same RNG, seed, EPOCH, property order, KEEP thresholds). It did **not** change
  puzzle output. The output-changing switch (CSV → on-the-fly) predates available
  git history.

## Verdict

Restoration is possible, but the source data is **outside this repo** — the repo
alone cannot regenerate the originals.

---

## Next steps (in order)

### 1. Check whether the originals are still in production KV (do this first)

Because KV is read-first and never overwritten, the puzzle objects written during
the CSV era are very likely **still present** in the prod `PUZZLES` namespace — in
which case the archive may already be correct, or restoration is trivial.

```bash
# Single spot check — puzzle #1
wrangler kv key get "2026-03-08" --binding PUZZLES

# Or via the Cloudflare dashboard: Workers & Pages → KV → PUZZLES namespace,
# look up keys 2026-03-08 .. 2026-04-01.
```

- **Key exists** → read it back. Compare its `answer`/`clues` against a fresh
  recompute for that date (see step 2). If KV matches what players saw, the
  archive is already accurate and there's nothing to restore — just document.
- **Key missing** → that date is being recomputed by the current method (wrong).
  Go to step 3.

### 2. Pinpoint exactly which dates diverge

Build a small throwaway script (Node, importing `src/worker/puzzle.ts`) that, for
each date from `2026-03-08` to ~`2026-04-01`:

- recomputes the puzzle via `runFilterLoop(makeRng(dateSeedInt(date)))`, and
- compares against the KV-stored object (dump KV with
  `wrangler kv key list --binding PUZZLES`).

Output a table of `date | puzzleNumber | kv.answer | recomputed.answer | match?`
so the affected range is unambiguous.

### 3. If KV is missing/incorrect — recover the original data

The old CSV / generator is not in this repo's history. Look for it in:

- a **pre-squash local clone** of the repo (check other machines / old checkouts),
- any **backup** of the puzzle CSV,
- Cloudflare deployment history / an old Pages build,
- the original `_worker.js` / `app.js` bundles from before the rewrite.

Once recovered, **backfill KV** for the affected dates only (never touch dates
that are already correct):

```bash
wrangler kv key put "2026-03-08" '<original puzzle JSON>' --binding PUZZLES
```

### 4. Wrap up

- Confirm `/api/puzzle/:num`, `/api/puzzle/:num/solution`, and `/puzzles` return
  the historically-accurate puzzle for the affected range.
- For any genuinely unrecoverable date, decide the fallback (leave recomputed, or
  flag the archive entry).
- Add a short note to `docs/ARCHITECTURE.md` recording the historical method
  change so this isn't re-discovered later.
