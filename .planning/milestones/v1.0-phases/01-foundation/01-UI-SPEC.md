## Color

### Token Palette

These are the 6 semantic tokens defined in CSS `@theme` (one custom property per token, light and dark values via Tailwind `dark:` variants).

| Token | Light value | Dark value | Role |
|-------|-------------|------------|------|
| `--color-bg` | `#FAFAFA` | `#121213` | Page background (60% dominant surface) |
| `--color-text` | `#262624` | `#FAF8F4` | Primary text |
| `--color-muted` | `rgba(38,38,36,0.7)` | `rgba(246,240,232,0.6)` | Secondary / subdued text |
| `--color-accent` | `#0A850A` | `#1EAD52` | Accent — see reserved list below |
| `--color-surface` | `#FFFFFF` | `#363634` | Input backgrounds, key backgrounds (30% secondary) |
| `--color-border` | `rgba(38,38,36,0.12)` | `rgba(246,240,232,0.1)` | Borders, dividers |

### 60/30/10 Split

| Role | Token | Covers |
|------|-------|--------|
| Dominant (60%) | `--color-bg` | Page background, screen shells |
| Secondary (30%) | `--color-surface` | Screen placeholder containers, footer area |
| Accent (10%) | `--color-accent` | Reserved elements listed below |

### Accent reserved for (exhaustive list for Phase 1)

- Focus ring on interactive elements (`outline-color: accent`)
- No other accent usage in Phase 1 — no buttons, no clue tags, no interactive elements are built in this phase

### Destructive

No destructive actions in Phase 1. No destructive token needed.

### Opacity modifier rule

Shadows, tints, and overlays use Tailwind opacity modifiers (e.g. `bg-accent/10`, `text-muted/60`). No separate tint tokens.

**Source:** D-01, D-02, D-03, D-04 (CONTEXT.md); STY-02, STY-03, STY-04, STY-05 (REQUIREMENTS.md); Claude's Discretion for exact accent hex values — current `#0A850A` / `#1EAD52` carried forward from `src/style.css` (verified starting point, must pass WCAG AA on `#FAFAFA` / `#121213` before Phase 2 ships interactive elements).

## Screen Architecture

### Screens

| Screen ID | Name | Initial visibility |
|-----------|------|--------------------|
| `welcome` | Welcome | Hidden (opacity-0, pointer-events-none) |
| `game` | Game | Hidden |
| `completion` | Completion | Hidden |

### State machine

- One screen is active at a time. Active screen has `opacity-1` and `pointer-events-auto`.
- All other screens are `opacity-0` and `pointer-events-none`.
- Transitions fire at 250ms ease (D-05).
- View Transition API used where `document.startViewTransition` is available. CSS opacity transition is the fallback. Both paths produce the same cross-fade visual (D-06).

### DOM pattern

- Each screen is a `<section data-screen="welcome|game|completion">` inside a `<main>` container.
- Active screen is tracked by a module-scoped `currentScreen` variable in a new `src/screens.ts` module.
- DOM cache follows the existing pattern: `const dom = { welcome: $('[data-screen="welcome"]'), ... }`.
- Data attributes drive selection, not class names (project convention from CONVENTIONS.md).

**Source:** SCR-01, SCR-02, SCR-03 (REQUIREMENTS.md); D-05, D-06 (CONTEXT.md); existing DOM cache pattern (CONTEXT.md code insights).

## CSS Coexistence

- New Tailwind entry point: `src/tailwind.css` with `@import 'tailwindcss'` and `@theme { ... }` token definitions.
- Old `src/style.css` remains untouched and loads in parallel from `index.html`.
- New screen containers (data-screen elements) use Tailwind classes only.
- Existing markup keeps its existing classes — no changes to legacy HTML.
- `src/theme.ts` continues toggling `.dark`/`.light` on `<html>`. Tailwind's `dark:` variant reads from the `dark` class on `html` (configured in `tailwind.config` as `darkMode: 'class'`).
- Old CSS is removed in Phase 6 only (D-07, D-08).

**Source:** D-07, D-08 (CONTEXT.md); DESIGN-SYSTEM.md.
