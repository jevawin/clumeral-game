## Spacing Scale

Declared values (multiples of 4, Tailwind built-in scale):

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px (p-1 / gap-1) | Icon gaps inside tags, clue digit gaps (2.5px native → round up to 4px in new markup), tag inner padding |
| sm | 8px (p-2 / gap-2) | Gap between clue tag cell and clue lines, gap between clue line 1 and line 2 |
| md | 16px (p-4 / gap-4) | Vertical gap between clues in the clue list, spacing below clue list before digit boxes |
| lg | 24px (p-6 / gap-6) | Content column horizontal padding, gap between digit boxes and keypad, gap between keypad and submit |
| xl | 32px (p-8 / gap-8) | Top padding of scrollable content below the sticky header |
| 2xl | 48px (py-12) | Not used in this phase |
| 3xl | 64px | Not used in this phase |

Exceptions:
- Digit boxes row: `h-28` (112px) total height — preserves `7rem` from existing `.digits` — sized to be tappable without using spacing tokens.
- Keypad buttons: `h-12` (48px) — minimum WCAG 2.5.8 touch target.
- Submit button: `h-14` (56px) — slightly taller than keypad keys to signal primary action; matches existing `3.25rem` (52px, close enough to `h-14`).
- Header height: `h-14` (56px) fixed sticky header.
- Content column: `max-w-sm` (384px) horizontally centred — matches existing `max-width: 30rem` convention.
- Hamburger touch target: minimum `h-10 w-10` (40px × 40px) — icon is 24×24, padded to 40×40.

**Source:** `src/style.css` (`.digits`, `.keypad__btn`, `.submit__btn` dimensions), DESIGN-SYSTEM.md (30rem max-width), REQUIREMENTS.md GAM-02.

## Color

All tokens defined in Phase 1 `src/tailwind.css`. Use Tailwind semantic class names — never hardcode hex.

| Role | Light value | Dark value | Tailwind class | Usage |
|------|-------------|------------|----------------|-------|
| Dominant (60%) | #FAFAFA | #121213 | `bg-bg` / `text-bg` | Game screen background, main content area |
| Secondary (30%) | #FFFFFF | #363634 | `bg-surface` | Digit boxes, keypad keys, history chips, menu dropdown panel |
| Accent (10%) | #0A850A | #1EAD52 | `text-accent` / `bg-accent` | See reserved list below |
| Muted | rgba(38,38,36,0.7) | rgba(246,240,232,0.6) | `text-muted` | Puzzle number in header, "Your guesses" history label, save-score label, clue type tag background tint |
| Text | #262624 | #FAF8F4 | `text-text` | Clue line 1, digit possibles, keypad digits (non-eliminated), submit button label (white override), menu item labels |
| Border | rgba(38,38,36,0.12) | rgba(246,240,232,0.1) | `border-border` | Header bottom border, digit box borders, keypad key borders, history chip borders, skeleton shimmer |

**Accent reserved for:**
1. Clue tag border and background tint (`color-mix(in srgb, accent 5%, transparent)` equivalent — use `bg-accent/5 border-accent`)
2. Clue tag text (the property type label)
3. Clue line 2 text (operator + value)
4. Active digit box border (when the keypad is open for that box)
5. Submit button background (primary CTA only)
6. Clue position indicator ("lit" state — the small squares showing which digit positions the clue applies to)
7. Correct-answer feedback text colour

**Eliminated state colours (not from token set — derive inline):**
- Eliminated keypad key: text opacity 25% (`text-text/25`), surface background stays, no shadow
- Eliminated digit in possibles grid: opacity 12% (`opacity-[0.12]`)

**Feedback semantic colours (hardcoded, not in token set — carry over from existing):**
- Correct feedback text: `text-[#1a7a3a] dark:text-[#4cc990]`
- Incorrect feedback text: `text-[#c03030] dark:text-[#f07070]`

**Accent reserved for:** Never use accent on menu items, borders, heading text, or any element not listed above.

**Source:** `src/tailwind.css` (token values), `src/style.css` (`.clue__tag`, `.clue__digit.lit`, `.digit-box.active`, `.submit__btn`, `.feedback--correct`, `.feedback--incorrect`), REQUIREMENTS.md STY-05.

## Layout Contract

### Game screen shell

```
[data-screen="game"]
  absolute inset-0 flex flex-col bg-bg
  overflow-y-auto
```

### Sticky header

```
<header data-game-header>
  h-14 sticky top-0 z-30
  flex items-center justify-between
  px-4 bg-bg
  border-b border-border
```

Header contents (left → centre → right):
- Left: `<span class="text-xl font-bold text-text">Clumeral</span>` + `<span class="text-muted text-base ml-2">#[N]</span>` (puzzle number beside logo)
- Right: `<button data-menu-btn>` — hamburger icon, `h-10 w-10 flex items-center justify-center`

**Source:** CONTEXT.md D-04, D-05, D-06 (sticky header, logo left, puzzle number, hamburger right, bottom border).

### Scrollable content column

```
<div data-game-content>
  flex-1 overflow-y-auto
  px-4 pt-8 pb-24      ← pb-24 keeps content above fixed footer
```

Inner column: `w-full max-w-sm mx-auto flex flex-col`

### Clue list

```
<div data-clue-list role="list" aria-label="Puzzle clues" aria-busy="true">
  display: grid
  grid-template-columns: max-content 1fr
  row-gap: md (16px)
  column-gap: sm (8px)   ← tightened from legacy 0.8rem
```

Each clue: `display: grid; grid-template-columns: subgrid; grid-column: 1 / -1`

No card wrapper. Clues render directly on `bg-bg`. (CONTEXT.md D-02, REQUIREMENTS.md GAM-01.)

### Skeleton loaders (loading state)

Three skeleton rows rendered by default; replaced by actual clues once the puzzle API responds. (CONTEXT.md D-03.)

Skeleton row: same grid layout as a real clue. Tag cell: `w-16 h-8 rounded bg-surface animate-pulse`. Lines cell: two lines, 80% and 40% width, `h-4 rounded bg-surface animate-pulse`.

### Digit boxes

```
<div data-digits class="flex gap-2.5 w-full mt-2.5 h-28" role="group" aria-label="Digit entry boxes">
  Three [data-digit="0|1|2"] divs, flex: 1 each
```

Each digit box:
- Default state: `bg-surface border-[1.5px] border-border rounded shadow-[3px_3px_0_rgba(38,38,36,0.12)] dark:shadow-[3px_3px_0_rgba(0,0,0,0.25)]`
- Active state (keypad open): add `border-accent shadow-[3px_3px_0_rgba(10,133,10,0.3)]`
- Correct state: `bg-[rgba(46,139,87,0.12)] border-[rgba(46,139,87,0.4)] pointer-events-none` — spans all three boxes when guess is correct

When unresolved: shows the `[data-digit-grid]` possibles grid (3×3 or 4-col layout, Inconsolata 500 16px).

When resolved (one digit remains): shows `[data-digit-resolved]` — Inconsolata 600 32px `text-text`.

### Keypad

```
<div data-keypad-wrap class="w-full mt-2.5 hidden [&.open]:block">
  <div data-keypad class="grid grid-cols-5 gap-[7px] w-full">
    10 buttons (0–9)
```

Each keypad button: `h-12 rounded-sm font-mono text-lg font-medium bg-surface border-[1.5px] border-border shadow-[2px_2px_0_rgba(38,38,36,0.12)] dark:shadow-[2px_2px_0_rgba(0,0,0,0.25)]`

Press feedback: `active:translate-x-[2px] active:translate-y-[2px] active:shadow-none`

## Component Inventory

### Sticky game header

- Element: `<header data-game-header>`
- Fixed position: `sticky top-0 z-30 h-14`
- Left: brand text + puzzle number

- Bottom border: `border-b border-border`
- Background: `bg-bg` (matches screen background, no separate colour)

### Clue tag

- Element: `<button class="..." data-clue-tag>`
- Font: Inconsolata 700 16px, uppercase, `tracking-wide`
- Border: 1.5px solid `border-accent`
- Background: `bg-accent/5`
- Text: `text-accent`
- Height: `h-[1.375rem]`
- Contains: tag text label + `[data-clue-tag-icon]` SVG (info icon, 14×14, `stroke-width-2.5`)
- On click: shows/hides tooltip popover for that clue's type

### Clue tag tooltip

- Element: `<div role="tooltip" data-tag-tip>`
- Position: `absolute bottom-full left-0 mb-2`
- Background: `bg-surface border border-border rounded-md shadow-md`
- Padding: `p-3`
- Contains: description text (`text-base text-text leading-snug`) + close button (`text-muted hover:text-text`)
- Min width: `min-w-[18rem]`

### Clue position indicators

- Element: `<div data-clue-digits class="flex justify-between gap-[2.5px]">` — 3 child `<span>` elements
- Each: `w-[1.375rem] h-[1.375rem] rounded-[1px] border`
- "Lit" (this position applies): `border-accent bg-accent/50`
- "Dim" (this position does not apply): `border-accent bg-accent/5`

### Digit box

- Element: `<div data-digit="0|1|2" role="button" tabindex="0">`
- See Layout Contract section for state variants.
- `aria-label="Hundreds digit"` / `"Tens digit"` / `"Units digit"`
- `aria-expanded` reflects whether keypad is open for this box

### Keypad

- Element: `<div data-keypad-wrap>` wrapping `<div data-keypad>`
- Buttons generated by `app.ts` at runtime — not hardcoded in HTML
- Each: `<button data-key="[0-9]" class="...">`
- Eliminated keys: `data-elim="true"` attribute for styling hook

### Submit button

- Element: `<button data-submit type="button">`
- Disabled when not all digit boxes have resolved to a single digit
- `aria-disabled` mirrors disabled state (not `disabled` attribute — prevents keyboard trap)

### Menu dropdown

- Element: `<div id="game-menu" data-menu role="dialog" aria-label="Game menu">`
- Positioned absolutely below the header, right-aligned
- `hidden` class removed on open; re-added on close
- Focus trap: on open, first item receives focus; Escape closes the menu

### Skeleton loader rows

- Element: `<div class="clue-skeleton" aria-hidden="true">` × 3
- Removed from DOM once `renderClues()` completes its first render
- Shimmer: `animate-pulse` on each placeholder shape

## Interaction Contract

| Interaction | Trigger | Response |
|-------------|---------|----------|
| Tap digit box | `click` / `keydown Enter/Space` on `[data-digit]` | Opens keypad for that box; sets box to `active` state; closes any previously open keypad |
| Tap keypad key | `click` on `[data-key]` | Toggles that digit's eliminated state in `possibles[box]`; re-renders digit box grid; marks key as `elim` if eliminated |
| Last digit remains | After any keypad tap | Closes keypad; renders resolved digit in box; updates `activeBox` to null |
| Submit visible | When all three boxes resolved | `[data-submit-wrap]` transitions from `hidden` to `visible`; submit button becomes tappable |
| Tap submit | `click` on `[data-submit]` | Disables button (sets `submitting = true`); POSTs to `/api/guess`; on response, renders feedback; on correct, triggers Phase 5 celebration |
| Tap hamburger | `click` on `[data-menu-btn]` | Opens `[data-menu]`; sets `aria-expanded="true"`; moves focus to first menu item |
| Tap outside menu | `click` on document outside `[data-menu]` | Closes menu; restores focus to hamburger button |
| Tap close button | `click` on `[data-menu-close]` | Closes menu; restores focus to hamburger button |
| Escape key | `keydown Escape` when menu open | Closes menu; restores focus to hamburger button |
| Tap dark mode toggle | `click` on `[data-theme-toggle]` in menu | Calls `initTheme()` toggle from `src/theme.ts`; updates icon + label |
| Tap archive link | `click` on archive `<a>` | Navigates to `/puzzles` (full page navigation) |
| Tap feedback trigger | `click` on `[data-fb-btn]` in menu | Closes menu; opens feedback modal (Phase 4) — in Phase 3, wire the click but modal content is Phase 4's scope |
| Tap HTP link | `click` on `[data-htp-btn]` in menu | Closes menu; opens how-to-play modal from `src/modals.ts` |
| Scroll on small screens | Scroll on `[data-game-content]` | Header stays pinned; content scrolls under it |
| Page load / puzzle fetch | On `loadPuzzle()` resolve | Replace skeleton rows with rendered clues; set `aria-busy="false"` on clue list |
| API error | On `loadPuzzle()` failure | Replace skeletons with error copy; no crash |

**Source:** CONTEXT.md D-07, D-08 (menu behaviour); REQUIREMENTS.md GAM-04, GAM-05; `src/app.ts` existing event flow.

## Accessibility Contract

- `<header>` is a landmark — no extra `role` needed
- Hamburger button: `aria-expanded` toggles true/false; `aria-controls="game-menu"`; visible label via `aria-label="Open menu"`
- Menu dropdown: `role="dialog"` with `aria-label="Game menu"`; focus trap while open; Escape closes
- Digit boxes: `role="button"` with `tabindex="0"` and `aria-label` per position; `aria-expanded` indicates keypad state
- Keypad buttons: standard `<button>` elements; eliminated keys retain `<button>` (not `disabled`) so screen readers can still reach them, but add `aria-pressed="true"` to indicate elimination
- Submit button: `aria-disabled="true"` when not yet enabled (not `disabled` attribute)
- Clue list: `role="list"` + `aria-label="Puzzle clues"` + `aria-busy="true"` during load, `aria-busy="false"` once rendered
- Feedback `<p>`: `aria-live="assertive"` — announces result immediately
- Clue tag tooltips: `role="tooltip"`, dismissed on Escape; `aria-describedby` wiring not required (tooltip is supplementary)
- Colour contrast: `text-text` on `bg-bg` passes WCAG AA; `text-white` on `bg-accent` passes WCAG AA; `text-text/25` for eliminated keys is intentionally low-contrast (visual decoration only — information not conveyed by colour alone via `aria-pressed`)
- Minimum touch targets: digit boxes `h-28` ÷ 3 positions = each digit box is full height of the row (112px × ~full width), keypad buttons `h-12` (48px), submit `h-14` (56px), hamburger `h-10 w-10` (40px), menu items `py-3` gives ≥ 44px row height

## Digit Elimination Visual Treatment

(Claude's Discretion per CONTEXT.md)

**Chosen approach: opacity fade, no strikethrough.**

In the possibles grid inside an open digit box: eliminated digits fade to `opacity-[0.12]`. This matches the existing implementation exactly and avoids the visual noise of strikethrough marks in a 3×3 grid where digits are already small.

In the keypad: eliminated keys fade their digit text to `text-text/25`. The key shape and border remain; only the digit fades. This preserves the key's spatial position (avoids layout shifts) and lets players visually track which digits are still available without needing to count.

Rationale: both match the current codebase, require no new visual vocabulary, and work cleanly in both light and dark mode.

**Source:** `src/style.css` `.elim` state on `.digit-box__grid span` and `.keypad__btn`.

## Dark Mode Toggle Style in Menu

(Claude's Discretion per CONTEXT.md)

**Chosen approach: icon swap with adjacent text label.**

The toggle shows the moon icon (`icon-moon`) when in light mode (tap to go dark) and the sun icon (`icon-sun`) when in dark mode (tap to go light). The label beside the icon reads "Dark mode" or "Light mode" accordingly. No toggle switch widget — the menu item itself IS the toggle, consistent with the existing implementation in the footer links.

Rationale: a labelled switch requires additional ARIA plumbing and is heavier than needed for a single toggle in a contextual menu. The icon-swap + label pattern matches the existing footer buttons and is immediately recognisable.
