# Project Research Summary

**Project:** David Lark's Lame Number Game
**Domain:** Static vanilla JS single-page browser puzzle game with CSV-driven filtering logic
**Researched:** 2026-03-07
**Confidence:** HIGH

## Executive Summary

David Lark's Lame Number Game is a deduction puzzle in the Wordle genre: the player is shown a set of filter-derived clues and must guess the 3-digit number (100–999) that satisfies all of them. The puzzle is generated by a filtering algorithm that iteratively narrows a dataset of pre-computed number attributes until exactly one candidate remains, then presents those filters as clues. This is a fully static product — no server, no database, no build step — with `data.csv` serving as the only data source, loaded at runtime via `fetch()` and deployed on GitHub Pages.

The recommended approach is a no-framework, no-bundler implementation using native ES Modules (`<script type="module">`), CSS custom properties for theming, and PapaParse (CDN) for robust CSV parsing. The architecture follows a strict unidirectional state flow: CSVLoader loads once on page startup, FilteringEngine is a pure function with no DOM access, GameController orchestrates state transitions, and a stateless Renderer writes to the DOM. This separation makes each component independently testable and prevents the "DOM as state" bugs that are endemic to small vanilla JS projects.

The primary risks are concentrated in the data layer: the actual `data.csv` has confirmed trailing spaces in 4 column headers, Google Sheets exports produce `\r\n` line endings that break naive CSV splitting, and the filtering algorithm can enter an infinite loop on pathological inputs (repeated-digit numbers like 111, 222). These are all mitigable with specific, low-cost defensive patterns — PapaParse for parsing, header `.trim()`, and a hard iteration cap on the filter loop — but they must be addressed in the first implementation phase, not retrofitted later.

## Key Findings

### Recommended Stack

The entire stack is CDN-delivered or native-browser — no npm, no build step, no `node_modules`. PapaParse 5.4.1 (cdnjs CDN) handles CSV parsing with correct RFC 4180 support, `dynamicTyping: true` converts numeric string values to numbers at parse time, and `download: true` mode wraps `fetch()` internally. ES Module `<script type="module">` provides file-level scoping and import/export without a bundler. CSS custom properties (`--*`) enable the frosted-glass dark theme at runtime without a preprocessor. Local development requires a local HTTP server (`python3 -m http.server 8080` or `npx serve .`) because `fetch()` is blocked on `file://` protocol.

**Core technologies:**
- HTML5 + single `index.html`: GitHub Pages serves the repo root directly; no configuration needed
- CSS3 custom properties + `backdrop-filter`: frosted glass cards matching n8n design reference; 95%+ browser support
- Vanilla JavaScript ES2022 + native ES Modules: project constraint; all required APIs (`fetch`, `async/await`, array methods) are natively supported
- PapaParse 5.4.1 via CDN: robust CSV parsing; handles `\r\n` line endings and type coercion transparently

**What not to use:** React/Vue/Svelte (require build step), Webpack/Vite/Rollup (violate no-build constraint), `XMLHttpRequest` (superseded), naive `split(',')` for CSV (silently corrupts data), `file://` for local testing (blocks fetch).

### Expected Features

The feature set is lean and dependency-ordered. Nothing in the UI layer is meaningful until the CSV loads and the filter loop produces a valid puzzle. Research identified 6 core features for v1 launch and 4 additive features for v1.x once the core is validated.

**Must have (table stakes):**
- CSV load with loading state and explicit error state — without data, there is no game
- Filtering algorithm that produces exactly one answer row — the core mechanic
- All clues displayed with human-readable labels — raw column IDs are unreadable
- 3-digit number input with Enter-to-submit and inline validation (100–999 range)
- Correct/incorrect feedback — the game loop payoff
- New Puzzle button — minimum replayability; requires full state reset (candidates, clues, input, phase)

**Should have (competitive differentiators):**
- Wrong-guess history — reinforces deduction; low complexity
- Clue explanation tooltips — reduces confusion about filter category names; minimal work, high payoff
- Share result copy-to-clipboard — social layer with no backend required
- Puzzle difficulty indicator (clue count) — sets expectation; already known at render time

**Defer (v2+):**
- Progressive clue reveal mode — conflicts with all-clues-at-once; requires UX redesign
- Daily puzzle mode — requires date-seeded RNG and data management contract
- Mobile layout optimization — explicitly deferred in PROJECT.md
- Multiple difficulty modes — doubles QA scope; difficulty emerges naturally from filter count variance

### Architecture Approach

The architecture is a 4-layer unidirectional system: Data Layer (data.csv + plain JS state object), Logic Layer (CSVLoader + FilteringEngine as pure modules), Game Controller (only component that writes to state and calls render), and Presentation Layer (stateless Renderer + thin event handlers). The project uses multiple files (`main.js`, `csvLoader.js`, `filterEngine.js`, `renderer.js`, `style.css`) rather than a single `index.html` because the filtering engine alone will exceed 100 lines and benefits from isolated testing. Build order is bottom-up: validate CSV schema first, implement CSVLoader, implement FilteringEngine, wire GameController, implement Renderer, then add UI event handlers last.

**Major components:**
1. CSVLoader (`csvLoader.js`) — owns all `fetch()` calls; wraps PapaParse; returns Promise of typed row objects; never called again after initial load
2. FilteringEngine (`filterEngine.js`) — pure function `runFilterLoop(rows)` that returns `{ answer, clues }`; no DOM, no global state; highest-risk component, build and stress-test first
3. GameController (`main.js`) — sole writer to the state object; calls CSVLoader once on startup; calls FilteringEngine on each new puzzle; calls `render(state)` after every state mutation
4. Renderer (`renderer.js`) — stateless DOM updater; reads state, writes HTML; never mutates state; replaces targeted elements, not the whole page
5. GameState — single plain JS object `{ phase, allRows, candidates, clues, answer }`; four phases: `loading`, `ready`, `solved`, `wrong`

### Critical Pitfalls

Seven pitfalls were identified; five are critical and must be addressed before the first end-to-end test.

1. **fetch() blocked on file:// protocol** — Use a local HTTP server from day one (`python3 -m http.server 8080`); never open `index.html` directly in browser; surface fetch errors visibly in UI, not just console.
2. **Trailing spaces in CSV column headers** — Confirmed in actual `data.csv` (cols 4, 12, 17, 22). Apply `.trim()` to every parsed header at parse time; assert all 23 headers are present after trimming.
3. **Infinite loop in filter algorithm** — Add a hard iteration cap (500 attempts) with a graceful escape path before running any end-to-end test; test specifically against repeated-digit numbers (111, 222, ... 999).
4. **Numeric type coercion** — PapaParse with `dynamicTyping: true` resolves this; without it, `"9" > "19"` evaluates true (string lexicographic order) causing silent wrong answers. Verify with `typeof` log after parse.
5. **GitHub Pages subdirectory path** — Use relative paths (`fetch('data.csv')`) not root-relative (`fetch('/data.csv')`); the latter resolves to the account root, not the repo subdirectory.

Lower priority but still required: `-webkit-backdrop-filter` prefix for Safari frosted glass support (visual QA before marking UI complete), and naive CSV split misalignment (mitigated by using PapaParse, but add column-count assertion as defense-in-depth).

## Implications for Roadmap

Based on the dependency chain (CSV must work before filtering, filtering must work before UI, UI must work before polish), a 4-phase structure is recommended. Each phase fully unlocks the next.

### Phase 1: Data Foundation
**Rationale:** Everything depends on the CSV parsing correctly. The confirmed trailing-space bug and type coercion risk must be resolved before any other code is written against the data. Attempting to build filtering or UI before this is validated invites silent, hard-to-diagnose bugs.
**Delivers:** Reliable, typed row data in memory; visible loading and error states; confirmation that all 23 columns are present and correctly named.
**Addresses:** CSV load with loading/error states (P1 table stakes); keyboard accessibility scaffolding via native `<input type="number">`
**Avoids:** Trailing spaces in headers (Pitfall 2), numeric type coercion bugs (Pitfall 7), naive CSV split misalignment (Pitfall 4), fetch() on file:// (Pitfall 1)

### Phase 2: Filtering Engine
**Rationale:** The FilteringEngine is the highest-risk, highest-value component. It is a pure function with no UI dependencies, so it can be built and thoroughly tested before any HTML is written. Stress-testing against edge cases (repeated-digit numbers) here prevents an infinite loop bug from shipping.
**Delivers:** `runFilterLoop(rows)` that reliably produces `{ answer, clues }` for any valid input; iteration cap with graceful escape; confirmed convergence on all 900 rows including pathological inputs.
**Uses:** FilteringEngine module pattern from ARCHITECTURE.md; pure function with no DOM access
**Avoids:** Infinite filter loop (Pitfall 3); sets the foundation that GameController depends on

### Phase 3: Game Loop and UI
**Rationale:** With data and logic confirmed working, the UI phase has no unknown risks. The state machine (loading → ready → solved/wrong → ready) is straightforward to implement once GameController wires CSVLoader and FilteringEngine together.
**Delivers:** Complete playable game — clue display with human-readable labels, guess input, correct/incorrect feedback, New Puzzle button with full state reset; dark theme with frosted-glass cards.
**Implements:** GameController, Renderer, GameState object, UI event handlers (all from ARCHITECTURE.md); CSS custom properties theming + backdrop-filter with -webkit- prefix (STACK.md)
**Avoids:** DOM-as-state anti-pattern, inline onclick handlers (module scope prevents them), mutating allRows during filtering, backdrop-filter Safari breakage (Pitfall 5)

### Phase 4: Deployment and Polish
**Rationale:** GitHub Pages deployment is last because it requires a working game to validate, and has its own pitfall (subdirectory path). Polish features (wrong-guess history, clue tooltips, share button) are additive and low-risk once the core loop is tested.
**Delivers:** Live GitHub Pages deployment; v1.x enhancements (wrong-guess history, clue explanation tooltips, share result copy-to-clipboard, puzzle difficulty indicator)
**Avoids:** GitHub Pages subdirectory 404 (Pitfall 6); validates relative asset paths before v1 is shared

### Phase Ordering Rationale

- Phase 1 before Phase 2: FilteringEngine receives typed row objects from CSVLoader; if types are wrong, every filter comparison is silently wrong. Validate the data contract first.
- Phase 2 before Phase 3: GameController calls `runFilterLoop()` synchronously; the UI cannot render clues that don't exist. The filter is the bottleneck.
- Phase 3 before Phase 4: Can't validate GitHub Pages paths or do UX polish until there is a working game to deploy and test.
- This ordering also matches ARCHITECTURE.md's explicit build order (data.csv validation → csvLoader.js → filterEngine.js → main.js → renderer.js → index.html/style.css → UI event handlers).

### Research Flags

Phases with standard, well-documented patterns (can skip `/gsd:research-phase`):
- **Phase 1 (Data Foundation):** CSV loading with PapaParse is thoroughly documented; patterns are directly specified in STACK.md and PITFALLS.md
- **Phase 3 (Game Loop and UI):** Unidirectional state flow with vanilla JS is an established pattern; CSS frosted glass with custom properties is documented in STACK.md
- **Phase 4 (Deployment):** GitHub Pages deploy from branch root is a 3-step process documented in official Pages docs; no research needed

Phases that may benefit from targeted research during planning:
- **Phase 2 (Filtering Engine):** The filter loop algorithm is ported from Google Sheets Apps Script. The column metadata (RANGE_GROUPS, which columns are numeric vs text, valid operators per column) needs to be fully mapped from the actual `data.csv` schema before implementation. This is data-discovery work, not internet research — inspect `data.csv` and the original Apps Script before writing the engine.

### Gaps to Address

- **Filter algorithm column metadata:** The full set of RANGE_GROUPS (which columns belong to each filter category, valid operators per category, whether numeric or text comparison) needs to be mapped from `data.csv` and the original Apps Script before Phase 2 implementation. This is the only significant unknown.
- **Human-readable clue label mapping:** FEATURES.md flags that column IDs must be mapped to plain-English labels. The exact mapping (column header → display label) needs to be defined before the Renderer is built. Low risk — this is a lookup table, not a logic question.
- **Convergence guarantee for all 900 rows:** The iteration cap prevents hangs, but it's not confirmed whether every number in the dataset can be uniquely identified by the available filters. If some numbers are indistinguishable, the escape path (re-roll or nearest candidate) needs a defined behavior. Discover this empirically in Phase 2 stress testing.
